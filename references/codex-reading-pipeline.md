# Codex精读管线 — 使用记录

## 成功案例1：子平真诠精读

**日期**: 2026-05-07
**模型**: QiHang gpt-5.4 via Codex CLI
**耗时**: ~10分钟
**Token**: 134K
**输入**: 子平真诠评注.txt (84KB, 48章)
**输出**: knowledge/子平真诠精读.md (3632行/192KB, v2深度版)

### 关键步骤

1. 确认目录是git repo（Codex必须在git目录运行）
2. `codex exec --full-auto` 后台启动
3. 提示词要求：先读已有进度→用grep -n找章节边界→逐章精读→每10章追加写入→更新PROGRESS.md
4. 用 `process(action='wait')` 监控，每60秒检查一次
5. 用 `wc -l` 验证产出文件确实增长

## 成功案例2：渊海子平+滴天髓并行精读

**日期**: 2026-05-07
**方法**: 多个 `codex exec --full-auto` 并行，每个处理源文件一个批次
**产出**: 渊海子平精读.md (2113行/114KB) + 滴天髓精读.md (2903行/137KB)

### 并行分批策略

源文件超过1000行时必须分批，每批对应源文件的一段行号范围：

**渊海子平 (2467行, 分3批):**
```
批1: 行1-460 → 基础十神
批2: 行461-1031 → 六亲赋文
批3: 行1032-2467 → 赋文格局
```

**滴天髓 (11590行, 分4批):**
```
批1: 行123-5334 → 通神论上半
批2: 行5335-end → 通神论下半+六亲论
批3/4: 按内容段落拆分
```

### 合并+清理流程

批文件全部完成后：
```bash
# 合并
cat 渊海子平_批1_*.md 渊海子平_批2_*.md 渊海子平_批3_*.md > knowledge/渊海子平精读.md

# 验证
wc -l knowledge/渊海子平精读.md  # 应该接近各批之和

# 清理批文件
rm 渊海子平_批*.md 滴天髓_批*.md
```

### ⚠️ 关键踩坑：codex exec默认不写文件（必须加 --full-auto）

`codex exec` 进程显示完成（exit code 0），但输出文件不存在。**根因是sandbox默认read-only。**

**修复**：必须加 `--full-auto` 参数。不加此参数的 `codex exec` 只能读文件、输出到stdout，不能写文件。

```bash
# ❌ 错误：sandbox read-only，不写文件
codex exec --model gpt-5.4 "读取X，精读，写入Y.md"

# ✅ 正确：--full-auto 绕过sandbox，可以写文件
codex exec --model gpt-5.4 --full-auto "读取X，精读，写入Y.md"
```

**其他对策**：
1. 每个批次的prompt中明确指定输出文件的**绝对路径**
2. 批次完成后立即用 `ls -la` 验证文件存在且大小合理（>10KB）
3. 如果文件不存在且没用 `--full-auto`，加上后重跑
4. `wc -l` 检查行数，预期每批至少300行；低于200行说明读取失败

### 并行度限制

- 最多同时4-5个codex进程（受API并发限制）
- 渊海子平3批 + 滴天髓4批 = 7个进程，分两波启动
- 第一波完成后再启动第二波，避免API限流

### Prompt模板

```
读取 [源文件路径] 的第 [start]-[end] 行。
按以下格式精读每一章：
- 原文金句（逐条引用）
- 核心规则（用大白话解释）
- 格局成败条件
- 案例拆解（如有）
- 例外和易错点
- 与《子平真诠》的异同对比

将精读结果写入 [绝对路径/输出文件.md]。
```

## 成功案例3：三命通会并行精读 ✅ 完成

**日期**: 2026-05-07
**模型**: QiHang gpt-5.4 via Codex CLI
**源文件**: knowledge/三命通会/卷一~卷十二.txt (12卷, 7737行, 1.3MB)
**产出**: 三命通会_精读.md (2520行/210KB)

### 关键修复：必须用 `--full-auto`

**踩坑**：前5轮 `codex exec` (无 `--full-auto`) 进程全部完成(exit 0)但零产出文件。
**根因**：codex exec默认sandbox为read-only，进程能读文件但不能写文件。
**修复**：`codex exec --model gpt-5.4 --full-auto "..."` — 加 `--full-auto` 后5个并行进程全部成功写入。

### 多卷合并分批策略（已验证）

```bash
cat 卷一.txt 卷二.txt 卷三.txt > /tmp/三命通会_批1.txt   # 1015行
cat 卷四.txt 卷五.txt 卷六.txt > /tmp/三命通会_批2.txt   # 1155行
cat 卷七.txt 卷八.txt > /tmp/三命通会_批3.txt             # 1898行
cat 卷九.txt 卷十.txt > /tmp/三命通会_批4.txt             # 2208行
cat 卷十一.txt 卷十二.txt > /tmp/三命通会_批5.txt         # 1461行
```

**并行启动命令**：
```bash
cd /path/to/skill-dir
codex exec --model gpt-5.4 --full-auto "读取 /tmp/三命通会_批N.txt，精读，写入 knowledge/三命通会_批N.md" > /tmp/codex_batchN.log 2>&1 &
```

**产出统计**：
| 批次 | 范围 | 行数 | 大小 |
|------|------|------|------|
| 批1 | 卷一至卷三 | 918 | 70KB |
| 批2 | 卷四至卷六 | 445 | 35KB |
| 批3 | 卷七至卷八 | 630 | 30KB |
| 批4 | 卷九至卷十 | 273 | 35KB |
| 批5 | 卷十一至卷十二 | 234 | 40KB |
| **合并** | **全部12卷** | **2520** | **210KB** |

### 进程log内容回收

当Codex进程完成但文件未写入时，notification payload中可能包含精读内容。
用 `process(action='log', process_id='xxx')` 可以捞出输出。
本案例中5个旧进程的notification包含完整精读内容，但因sandbox限制未写入文件。
新进程（加 `--full-auto`）重新执行后成功写入。
